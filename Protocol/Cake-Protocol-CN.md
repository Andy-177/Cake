# Cake-Protocol 协议规范

# 1. 协议概述

Cake-Protocol 是一种基于 TCP 传输的应用层二进制通信协议，专为客户端-服务器架构设计，支持点对点消息、广播消息和群组消息功能，并内置心跳保活机制，保证连接的稳定性。

## 1.1 协议设计目标

- 轻量级：二进制格式，解析高效，开销小

- 可靠：基于 TCP 保证消息有序、不丢失

- 易扩展：固定包头+可变包体结构，便于新增消息类型

- 高可用：内置心跳机制，支持连接状态检测

## 1.2 核心特性

- 客户端ID分配：服务器为每个连接客户端分配唯一8字节ID

- 点对点通信：客户端间基于ID的定向消息传输

- 广播通信：向所有在线客户端发送消息

- 群组通信：支持创建/注销群组、群组内消息广播

- 心跳保活：定时心跳检测，自动断开异常连接

# 2. 基础数据格式

## 2.1 字节序

所有多字节数据均采用 **网络字节序（大端序，Big-Endian）**

# 3. 数据包通用结构

所有Cake-Protocol数据包均遵循「固定包头 + 可变包体」结构：

|字段|类型|字节长度|说明|
|---|---|---|---|
|PacketType|uint8|1|数据包类型（见3.1）|
|BodyLength|uint32|4|包体部分的字节长度|
|Body|bytes[]|可变|数据包体（根据类型不同）|
**包头总长度固定为5字节（1+4）**，包体长度由BodyLength字段指定，取值范围：0 ≤ BodyLength ≤ 65535（建议最大值4096）。

## 3.1 数据包类型定义（PacketType）

|类型值|名称|英文标识|用途|
|---|---|---|---|
|0x01|ID请求包|ID_REQUEST|客户端请求服务器分配ID|
|0x02|ID响应包|ID_RESPONSE|服务器返回客户端唯一ID|
|0x03|点对点/广播业务包|BUSINESS|发送点对点/广播消息|
|0x04|心跳包|HEARTBEAT|心跳检测（请求/响应）|
|0x05|群组注册包|GROUP_REGISTER|客户端请求创建群组|
|0x06|群组业务包|GROUP_BUSINESS|发送群组消息|
|0x07|群组ID响应包|GROUP_RESPONSE|服务器返回创建的群组ID|
|0x08|群组注销包|GROUP_UNREGISTER|客户端请求注销群组|
# 4. 具体数据包格式

## 4.1 ID请求包（0x01）

- **用途**：客户端连接成功后，向服务器请求分配唯一ID

- **PacketType**：0x01

- **BodyLength**：0（无包体）

- **包体**：空

- **示例**：

```plain text
01 00 00 00 00
```

## 4.2 ID响应包（0x02）

- **用途**：服务器为客户端分配8字节唯一ID并返回

- **PacketType**：0x02

- **BodyLength**：8

- **包体结构**：

|字段|类型|字节长度|说明|
|---|---|---|---|
|ClientID|bytes[8]|8|客户端唯一ID|
**示例**：

```plain text
02 00 00 00 08 a1 b2 c3 d4 e5 f6 78 90
```

（其中 a1 b2 c3 d4 e5 f6 78 90 为分配的客户端ID）

## 4.3 点对点/广播业务包（0x03）

- **用途**：客户端发送点对点消息或广播消息

- **PacketType**：0x03

- **BodyLength**：16 + 消息长度（8+8+消息字节数）

- **包体结构**：

|字段|类型|字节长度|说明|
|---|---|---|---|
|SrcID|bytes[8]|8|发送方客户端ID|
|DestID|bytes[8]|8|接收方ID（广播时为全0xff）|
|Message|bytes[]|可变|消息内容（UTF-8编码）|
**示例**：

发送方ID：a1 b2 c3 d4 e5 f6 78 90，接收方ID：ff ff ff ff ff ff ff ff（广播），消息：hello

```plain text
03 00 00 00 15 a1 b2 c3 d4 e5 f6 78 90 ff ff ff ff ff ff ff ff 68 65 6c 6c 6f
```

（BodyLength=21，16+5=21）

## 4.4 心跳包（0x04）

- **用途**：客户端定时发送心跳包，服务器收到后返回相同心跳包作为响应

- **PacketType**：0x04

- **BodyLength**：0（无包体）

- **包体**：空

- **示例**：

```plain text
04 00 00 00 00
```

## 4.5 群组注册包（0x05）

- **用途**：客户端请求创建群组，指定群组成员列表

- **PacketType**：0x05

- **BodyLength**：8 × 成员数量

- **包体结构**：

|字段|类型|字节长度|说明|
|---|---|---|---|
|MemberIDs|bytes[8]*N|8×N|N个群组成员的客户端ID|
**示例**：

成员1ID：a1 b2 c3 d4 e5 f6 78 90，成员2ID：01 02 03 04 05 06 07 08

```plain text
05 00 00 00 10 a1 b2 c3 d4 e5 f6 78 90 01 02 03 04 05 06 07 08
```

（BodyLength=16，8×2=16）

## 4.6 群组业务包（0x06）

- **用途**：客户端向指定群组发送消息

- **PacketType**：0x06

- **BodyLength**：16 + 消息长度（8+8+消息字节数）

- **包体结构**：

|字段|类型|字节长度|说明|
|---|---|---|---|
|SrcID|bytes[8]|8|发送方客户端ID|
|GroupID|bytes[8]|8|目标群组ID|
|Message|bytes[]|可变|消息内容（UTF-8编码）|
**示例**：

发送方ID：a1 b2 c3 d4 e5 f6 78 90，群组ID：99 88 77 66 55 44 33 22，消息：hi group

```plain text
06 00 00 00 19 a1 b2 c3 d4 e5 f6 78 90 99 88 77 66 55 44 33 22 68 69 20 67 72 6f 75 70
```

（BodyLength=25，16+9=25）

## 4.7 群组ID响应包（0x07）

- **用途**：服务器为客户端创建群组后，返回8字节唯一群组ID

- **PacketType**：0x07

- **BodyLength**：8

- **包体结构**：

|字段|类型|字节长度|说明|
|---|---|---|---|
|GroupID|bytes[8]|8|群组唯一ID|
**示例**：

```plain text
07 00 00 00 08 99 88 77 66 55 44 33 22
```

## 4.8 群组注销包（0x08）

- **用途**：客户端请求注销指定群组

- **PacketType**：0x08

- **BodyLength**：8

- **包体结构**：

|字段|类型|字节长度|说明|
|---|---|---|---|
|GroupID|bytes[8]|8|待注销群组ID|
**示例**：


# 5. 通信流程规范

## 5.1 客户端连接流程

1. 客户端创建TCP socket，连接服务器指定IP和端口

2. 客户端发送ID请求包（0x01）

3. 服务器接收后，分配8字节唯一ClientID，返回ID响应包（0x02）

4. 客户端解析并保存ClientID，取消socket超时设置

5. 客户端启动心跳线程（定时发送心跳包）和接收线程（监听服务器消息）

## 5.2 心跳保活流程

1. 客户端每隔N秒（建议5秒）发送心跳包（0x04）

2. 服务器收到心跳包后，立即返回相同的心跳包作为响应

3. 客户端记录最后一次收到心跳响应的时间

4. 若客户端超过M秒（建议20秒）未收到心跳响应，判定连接异常，主动断开TCP连接

5. 服务器若超过M秒未收到客户端心跳包，主动断开TCP连接

## 5.3 点对点消息发送流程

1. 发送方客户端构造点对点业务包（0x03），DestID填写接收方ClientID

2. 发送方将数据包发送至服务器

3. 服务器解析数据包，根据DestID找到对应在线客户端

4. 服务器将数据包转发至接收方客户端

5. 接收方客户端解析数据包，提取并展示消息内容

## 5.4 广播消息发送流程

1. 发送方客户端构造广播业务包（0x03），DestID填写全0xff

2. 发送方将数据包发送至服务器

3. 服务器解析数据包，识别为广播消息，转发至所有在线客户端

4. 所有在线客户端解析并展示广播消息

## 5.5 群组创建与消息流程

### 5.5.1 群组创建

1. 客户端构造群组注册包（0x05），填写群组成员ClientID列表

2. 客户端发送注册包至服务器

3. 服务器创建群组，分配8字节唯一GroupID，返回群组ID响应包（0x07）

4. 客户端保存GroupID，完成群组注册

### 5.5.2 群组消息发送

1. 客户端构造群组业务包（0x06），填写发送方ClientID和目标GroupID

2. 客户端发送群组业务包至服务器

3. 服务器解析数据包，根据GroupID找到群组成员列表

4. 服务器将数据包转发至所有群组成员客户端

5. 群组成员客户端解析并展示群组消息

### 5.5.3 群组注销

1. 客户端构造群组注销包（0x08），填写待注销GroupID

2. 客户端发送注销包至服务器

3. 服务器删除该群组信息，完成注销

# 6. 异常处理规范

## 6.1 数据包解析异常

- 若包头长度不足5字节，丢弃当前数据，等待后续数据拼接

- 若BodyLength与实际包体长度不符，丢弃该数据包，记录异常日志

- 若PacketType为未知类型，丢弃该数据包，记录异常日志

## 6.2 连接异常

- 客户端未收到ID响应包（超时10秒）：断开连接，提示连接失败

- 心跳超时：客户端/服务器主动断开TCP连接

- 发送消息时连接异常：关闭客户端，清理资源

## 6.3 业务异常

- 发送消息时ClientID未分配：拒绝发送，提示未连接服务器

- 群组注册时成员ID格式错误：返回注册失败，提示格式错误

- 发送群组消息时群组不存在：拒绝转发，提示群组已注销/不存在

# 7. 协议扩展规范

## 7.1 新增数据包类型

- 新增PacketType值需从0x09开始递增，避免与现有类型冲突

- 新增数据包需遵循「固定包头+可变包体」结构

- 新增类型需在协议文档中明确说明用途、包体结构和交互流程

## 7.2 字段扩展

- 若需扩展现有包体字段，建议在包体末尾新增，保持原有字段顺序不变

- 扩展后需更新BodyLength计算规则，保证兼容性
