# Cake-Protocol プロトコル仕様

# 1. プロトコル概要

Cake-Protocol は、TCP ベースのアプリケーション層バイナリ通信プロトコルで、クライアント・サーバーアーキテクチャ向けに設計されています。
ピアツーピアメッセージ、ブロードキャストメッセージ、グループメッセージ機能をサポートし、ハートビートによるキープアライブ機構を内蔵し、接続の安定性を確保します。

## 1.1 プロトコル設計目標

- 軽量：バイナリ形式で解析効率が高く、オーバーヘッドが小さい
- 信頼性：TCP によりメッセージの順序保証・欠落防止を実現
- 拡張性：固定ヘッダー + 可変ボディ構造により、新たなメッセージタイプの追加が容易
- 高可用性：ハートビート機構を内蔵し、接続状態の検出をサポート

## 1.2 コア機能

- クライアントID 割り当て：サーバーが各接続クライアントに一意な 8 バイト ID を割り当て
- ピアツーピア通信：ID に基づくクライアント間の定向メッセージ転送
- ブロードキャスト通信：全オンラインクライアントへのメッセージ送信
- グループ通信：グループの作成・削除、グループ内ブロードキャストをサポート
- ハートビートキープアライブ：定期的なハートビート監視、異常接続の自動切断

# 2. 基本データ形式

## 2.1 バイトオーダー

# 3. パケット共通構造

すべての Cake-Protocol パケットは**固定ヘッダー + 可変ボディ**の構造に従います。

| フィールド | 型 | バイト長 | 説明 |
|------------|------|----------|------|
| PacketType | uint8 | 1 | パケット種別（3.1参照） |
| BodyLength | uint32 | 4 | パケットボディのバイト長 |
| Body | bytes[] | 可変 | パケットボディ（種別により内容が異なる） |

**ヘッダー全体の長さは固定で 5 バイト（1+4）**。
ボディ長は BodyLength フィールドで指定し、範囲：0 ≤ BodyLength ≤ 65535（推奨最大値 4096）。

# 4. 個別パケット形式

## 4.1 ID要求パケット（0x01）

- **用途**：クライアントが接続確立後、サーバーに一意IDの割り当てを要求
- **PacketType**：0x01
- **BodyLength**：0（ボディなし）
- **ボディ**：空
- **例**：
```plain text
01 00 00 00 00
```

## 4.2 ID応答パケット（0x02）

- **用途**：サーバーがクライアントに 8 バイトの一意IDを割り当てて返却
- **PacketType**：0x02
- **BodyLength**：8
- **ボディ構造**：

| フィールド | 型 | バイト長 | 説明 |
|------------|------|----------|------|
| ClientID | bytes[8] | 8 | クライアント一意ID |

- **例**：
```plain text
02 00 00 00 08 a1 b2 c3 d4 e5 f6 78 90
```
（a1 b2 c3 d4 e5 f6 78 90 が割り当てられたクライアントID）

## 4.3 ピアツーピア/ブロードキャスト業務パケット（0x03）

- **用途**：クライアントがピアツーピアメッセージまたはブロードキャストメッセージを送信
- **PacketType**：0x03
- **BodyLength**：16 + メッセージ長（8+8+メッセージバイト数）
- **ボディ構造**：

| フィールド | 型 | バイト長 | 説明 |
|------------|------|----------|------|
| SrcID | bytes[8] | 8 | 送信元クライアントID |
| DestID | bytes[8] | 8 | 送信先ID（ブロードキャスト時は全 0xff） |
| Message | bytes[] | 可変 | メッセージ内容（UTF-8 エンコード） |

- **例**：
送信元ID：a1 b2 c3 d4 e5 f6 78 90、送信先ID：ff ff ff ff ff ff ff ff（ブロードキャスト）、メッセージ：hello
```plain text
03 00 00 00 15 a1 b2 c3 d4 e5 f6 78 90 ff ff ff ff ff ff ff ff 68 65 6c 6c 6f
```
（BodyLength=21、16+5=21）

## 4.4 ハートビートパケット（0x04）

- **用途**：クライアントが定期的にハートビートを送信、サーバーは受信後に同一パケットを応答
- **PacketType**：0x04
- **BodyLength**：0（ボディなし）
- **ボディ**：空
- **例**：
```plain text

```

## 4.5 グループ登録パケット（0x05）

- **用途**：クライアントがグループ作成を要求、メンバーリストを指定
- **PacketType**：0x05
- **BodyLength**：8 × メンバー数
- **ボディ構造**：

| フィールド | 型 | バイト長 | 説明 |
|------------|------|----------|------|
| MemberIDs | bytes[8]*N | 8×N | N 個のクライアントID（グループメンバー） |

- **例**：
メンバー1ID：a1 b2 c3 d4 e5 f6 78 90、メンバー2ID：01 02 03 04 05 06 07 08
```plain text
05 00 00 00 10 a1 b2 c3 d4 e5 f6 78 90 01 02 03 04 05 06 07 08
```
（BodyLength=16、8×2=16）

## 4.6 グループ業務パケット（0x06）

- **用途**：クライアントが指定グループにメッセージを送信
- **PacketType**：0x06
- **BodyLength**：16 + メッセージ長（8+8+メッセージバイト数）
- **ボディ構造**：

| フィールド | 型 | バイト長 | 説明 |
|------------|------|----------|------|
| SrcID | bytes[8] | 8 | 送信元クライアントID |

- **例**：
送信元ID：a1 b2 c3 d4 e5 f6 78 90、グループID：99 88 77 66 55 44 33 22、メッセージ：hi group

（BodyLength=25、16+9=25）

# 5. 通信フロー仕様

## 5.1 クライアント接続フロー

1. クライアントが TCP ソケットを作成し、サーバーの指定IP・ポートに接続
2. クライアントが ID要求パケット（0x01）を送信
3. サーバーが受信後、8 バイトの一意 ClientID を割り当て、ID応答パケット（0x02）を返却
4. クライアントが ClientID を解析・保存し、ソケットのタイムアウトを無効化
5. クライアントがハートビートスレッド（定期送信）と受信スレッド（サーバーメッセージ監視）を起動

## 5.2 ハートビートキープアライブフロー

1. クライアントが N 秒ごと（推奨 5 秒）にハートビートパケット（0x04）を送信
2. サーバーはハートビートを受信後、即座に同一パケットを応答
3. クライアントは最後にハートビート応答を受信した時刻を記録
4. クライアントが M 秒以上（推奨 20 秒）応答を受信しない場合、接続異常と判断して TCP を切断
5. サーバーが M 秒以上クライアントからのハートビートを受信しない場合、TCP を切断

## 5.3 ピアツーピアメッセージ送信フロー

1. 送信元クライアントがピアツーピア業務パケット（0x03）を構築、DestID に受信者の ClientID を設定
2. 送信元がパケットをサーバーに送信
3. サーバーがパケットを解析し、DestID から対象のオンラインクライアントを検索
4. サーバーがパケットを受信者に転送
5. 受信者クライアントがパケットを解析し、メッセージを抽出・表示

## 5.4 ブロードキャストメッセージ送信フロー

1. 送信元クライアントがブロードキャスト業務パケット（0x03）を構築、DestID に全 0xff を設定
2. 送信元がパケットをサーバーに送信
3. サーバーがパケットを解析し、ブロードキャストと認識して全オンラインクライアントに転送
4. 全オンラインクライアントがパケットを解析・表示

## 5.5 グループ作成・メッセージフロー

### 5.5.1 グループ作成

1. クライアントがグループ登録パケット（0x05）を構築、メンバー ClientID リストを設定
2. クライアントがサーバーに送信
3. サーバーがグループを作成し、8 バイトの一意 GroupID を割り当て、グループID応答パケット（0x07）を返却
4. クライアントが GroupID を保存し、登録完了

### 5.5.2 グループメッセージ送信

1. クライアントがグループ業務パケット（0x06）を構築、送信元 ClientID と対象 GroupID を設定
2. クライアントがサーバーに送信
3. サーバーがパケットを解析し、GroupID からメンバーリストを取得
4. サーバーがパケットを全グループメンバーに転送
5. グループメンバーがパケットを解析・表示

### 5.5.3 グループ削除

1. クライアントがグループ削除パケット（0x08）を構築、削除対象 GroupID を設定
2. クライアントがサーバーに送信
3. サーバーが当該グループ情報を削除し、処理完了

# 6. 例外処理仕様

## 6.1 パケット解析例外

- ヘッダー長が 5 バイトに満たない場合：現在のデータを破棄し、後続データの結合を待つ
- BodyLength が実際のボディ長と一致しない場合：パケットを破棄し、異常ログを記録
- PacketType が未知の場合：パケットを破棄し、異常ログを記録

## 6.2 接続例外

- 10 秒以内に ID応答を受信しない場合：切断し、接続失敗を通知
- ハートビートタイムアウト：クライアント・サーバーが自発的に TCP を切断
- メッセージ送信中に接続異常：クライアントを終了し、リソースを解放

## 6.3 業務例外

- ClientID 割り当て前にメッセージを送信した場合：送信を拒否し、未接続を通知
- グループ登録時にメンバーIDの形式が不正な場合：登録失敗を返却し、形式エラーを通知
- 存在しないグループにメッセージを送信した場合：転送を拒否し、グループが削除済み・未存在であることを通知
